The Measurements service provides methods for creating and managing measurement requests across data providers.

## Resource

### Measurement

A measurement from a set of `DataProvider`s requested by a `MeasurementConsumer`.

**Resource name pattern:** `measurementConsumers/{measurement_consumer}/measurements/{measurement}`

<ResponseField name="name" type="string" required>
  Resource name identifier.
</ResponseField>

<ResponseField name="measurement_consumer_certificate" type="string" required>
  Resource name of the `Certificate` belonging to the parent `MeasurementConsumer`.
</ResponseField>

<ResponseField name="measurement_spec" type="SignedMessage" required>
  Serialized `MeasurementSpec` for requisitions, which can be verified using `measurement_consumer_certificate`.
  
  The version of the `MeasurementSpec` message type must correspond to the API version that this resource was created in.
</ResponseField>

<ResponseField name="data_providers" type="DataProviderEntry[]" required>
  Map of `DataProvider` name to parameters for that `DataProvider`.
</ResponseField>

<ResponseField name="protocol_config" type="ProtocolConfig">
  The `ProtocolConfig` selected for this measurement according to the `measurement_spec`. Output only.
</ResponseField>

<ResponseField name="state" type="State">
  State of this `Measurement`. Output only.
  
  Possible values:
  - `STATE_UNSPECIFIED` - Default value used if the state is omitted
  - `AWAITING_REQUISITION_FULFILLMENT` - Waiting for all linked `Requisition`s to be fulfilled
  - `COMPUTING` - Computation is running
  - `SUCCEEDED` - Completed successfully (terminal state)
  - `FAILED` - Completed with failure (terminal state)
  - `CANCELLED` - Cancelled by Measurement Consumer (terminal state)
</ResponseField>

<ResponseField name="results" type="ResultOutput[]">
  Results of this `Measurement`. Only set if `state` is `SUCCEEDED`. Output only.
</ResponseField>

<ResponseField name="measurement_reference_id" type="string">
  ID referencing the `Measurement` in an external system, provided by the `MeasurementConsumer`.
</ResponseField>

<ResponseField name="failure" type="Failure">
  Information about the failure of this `Measurement`. Set when the `state` is set to `FAILED`. Output only.
</ResponseField>

<ResponseField name="create_time" type="Timestamp">
  When the `Measurement` was created. Output only.
</ResponseField>

<ResponseField name="update_time" type="Timestamp">
  When the `Measurement` was last updated. Output only.
</ResponseField>

### DataProviderEntry

Entry for `data_providers` map.

<ResponseField name="key" type="string" required>
  Key of the map entry, which is a `DataProvider` resource name.
</ResponseField>

<ResponseField name="value" type="Value" required>
  Value of the map entry.
  
  <Expandable title="Value fields">
    <ResponseField name="data_provider_certificate" type="string" required>
      Resource name of the `Certificate` belonging to `data_provider`.
    </ResponseField>
    
    <ResponseField name="data_provider_public_key" type="Any" required>
      Pre-shared packed `EncryptionPublicKey`. The version of the `EncryptionPublicKey` message type must correspond to the API version that this resource was created in.
    </ResponseField>
    
    <ResponseField name="encrypted_requisition_spec" type="EncryptedMessage" required>
      Encrypted `SignedMessage` containing the serialized `RequisitionSpec` for this entry, which can be verified using `measurement_consumer_certificate`. The encryption uses `data_provider_public_key` as the recipient public key.
    </ResponseField>
    
    <ResponseField name="nonce_hash" type="bytes" required>
      SHA256 hash of the `nonce` from `encrypted_requisition_spec`, where the nonce value has big-endian byte ordering.
    </ResponseField>
  </Expandable>
</ResponseField>

</ResponseField>

### Failure

Information about a failure.

<ResponseField name="reason" type="Reason" required>
  Reason for this `Failure`.
  
  Possible values:
  - `REASON_UNSPECIFIED` - Default value used if the reason is omitted
  - `CERTIFICATE_REVOKED` - An associated certificate was revoked
  - `REQUISITION_REFUSED` - Requisition state was set to REFUSED
  - `COMPUTATION_PARTICIPANT_FAILED` - ComputationParticipant state was set to FAILED
</ResponseField>

<ResponseField name="message" type="string">
  Human-readable message. This should not contain any sensitive information.
</ResponseField>

**Request**

<ParamField query="page_size" type="int32">

</ParamField>

  When paginating, all other parameters provided to `ListMeasurementsRequest` must match the call that provided the page token.
</ParamField>

<ParamField body="filter" type="Filter">

    <ParamField name="updated_after" type="Timestamp">

    </ParamField>
    
    <ParamField name="updated_before" type="Timestamp">
      Filter for measurements that are updated before a certain timestamp.
    </ParamField>
    
    <ParamField name="created_before" type="Timestamp">
      Filter for measurements that are created before a certain timestamp.

## State Transitions

Measurements follow a specific state transition workflow:

```
  [Created]
      |
      v
AWAITING_REQUISITION_FULFILLMENT
      |
      +---> CANCELLED (terminal)
      |
      v
  COMPUTING
      |
      +---> FAILED (terminal)
      +---> CANCELLED (terminal)
      |
      v
  SUCCEEDED (terminal)
```

### State Descriptions

- **AWAITING_REQUISITION_FULFILLMENT**: The measurement is waiting for all linked requisitions to be fulfilled by data providers.
- **COMPUTING**: All requisitions have been fulfilled and the computation is running.
- **SUCCEEDED**: The measurement completed successfully and results are available.
- **FAILED**: The measurement failed due to an error (e.g., certificate revoked, requisition refused, computation participant failed).
- **CANCELLED**: The measurement was cancelled by the measurement consumer.

## Workflow

1. **Create Measurement**: A MeasurementConsumer creates a measurement specifying the data providers and measurement specifications.
2. **Requisition Creation**: The system automatically creates requisitions for each data provider.
3. **Await Fulfillment**: The measurement enters `AWAITING_REQUISITION_FULFILLMENT` state while waiting for data providers.
4. **Data Collection**: Data providers fulfill requisitions by providing the requested data.
5. **Computation**: Once all requisitions are fulfilled, the measurement transitions to `COMPUTING` state.
6. **Result**: The measurement transitions to either `SUCCEEDED` with results or `FAILED` with failure information.
